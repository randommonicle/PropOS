# PropOS Cross-Phase Audit — 2026-05-10

**End of Phase 3, pre-Phase-4.** Statutory-citation consistency + architectural drift audit covering all completed phases (1a–1i.3). Sister document to `SECURITY_AUDIT_2026-05-10.md`: that doc surfaced 38 RLS / CHECK / trigger findings at the end of 1h.3 and fed 1i.1 (12 closed). This doc surfaces 31 findings concentrated in the **regulatory-citation drift** that accumulated across 1a–1h.3 and the **RLS-vs-UI gate divergences** that surfaced when 1i.3 widened `is_pm_or_admin()`.

Scope is "lexical consistency" (DECISIONS 2026-05-10 — Audit scope decision). Regulatory correctness (whether a cited rule actually says what it claims to) is explicitly out of scope here.

## Executive summary

**31 findings.** Critical regulatory: 3 · Tier-1 regulatory / load-bearing: 9 · Tier-2 architectural debt: 11 · Tier-3 cleanup / opportunistic: 8.

**The 3 critical regulatory findings are one attack chain:**

When 1i.3 widened `is_pm_or_admin()` to include `accounts` and `senior_pm` (correct for the new write-tier semantics), the RLS write policies on `payment_authorisations`, `bank_accounts`, `contractors`, and `invoices` inherited the widening — but the **self-auth guard and the payee-setter ≠ release-authoriser segregation gate are enforced application-side only** in `PaymentAuthorisationsTab.handleAuthorise`. An `accounts` user with direct `supabase-js` access can chain:

1. INSERT a `payment_payee_setup` PA (legitimate)
2. UPDATE the PA to `status='authorised', authorised_by=<self>` — RLS permits (self-auth check is app-side; no `WITH CHECK requested_by IS DISTINCT FROM auth.uid()`)
3. UPDATE `contractors.approved_by=<self>, approved=true` — RLS permits (no column-level grant)
4. INSERT + self-authorise a `payment_release` to the same contractor

End state: money moves with a single staff member's involvement, defeating RICS Client money handling — segregation of duties; both signatories must be staff of the firm.

**The single highest-leverage fix is one line of SQL:** add `requested_by IS DISTINCT FROM auth.uid()` to the `WITH CHECK` of the `payment_auth_update` policy. Closes the self-auth half of the chain (steps 2 + 4) at the RLS layer. Independent of the financial-rules Edge Function commit. See finding **A-1**.

**The dominant non-critical theme is statutory-citation drift.** The 1d / 1e / 1g.5 era code is still on the deprecated "RICS Client Money Rule 4.7" framing; new code (1i.3, 1h reconciliation, 1i.2 invoices) uses the canonical "RICS Client money handling — segregation of duties; both signatories must be staff of the firm" per memory rule (`project_finance_roles_admin_only.md`). One commit canonicalises the lot across 4 components + 4 smokes.

## 1. Critical findings — concrete RLS-vs-UI gate divergence

### A-1 · `payment_authorisations` UPDATE permits self-authorisation at RLS layer

- **Severity:** CRITICAL
- **File:** `supabase/migrations/00026_security_hardening.sql:255-258`
- **Current:**
  ```sql
  CREATE POLICY payment_auth_update ON public.payment_authorisations
    FOR UPDATE
    USING      (firm_id = auth_firm_id() AND is_pm_or_admin())
    WITH CHECK (firm_id = auth_firm_id() AND is_pm_or_admin());
  ```
- **Issue:** `PaymentAuthorisationsTab.handleAuthorise` rejects self-authorisation at lines 147 (`pa.requested_by === userId`) — the UI gate. RLS has no such predicate. Post-1i.3, `is_pm_or_admin()` widens to admin / accounts / senior_pm / property_manager, so any of those four can `supabase.from('payment_authorisations').update({ status: 'authorised', authorised_by: <self> }).eq('id', <their-own-pa>)` and the self-auth check is silently bypassed. RICS Client money handling — segregation of duties (both signatories must be distinct staff) — defeated.
- **Fix:** New migration `00030_payment_auth_self_auth_rls.sql`:
  ```sql
  DROP POLICY IF EXISTS payment_auth_update ON public.payment_authorisations;
  CREATE POLICY payment_auth_update ON public.payment_authorisations
    FOR UPDATE
    USING      (firm_id = auth_firm_id() AND is_pm_or_admin())
    WITH CHECK (firm_id = auth_firm_id()
                AND is_pm_or_admin()
                AND requested_by IS DISTINCT FROM auth.uid());
  ```
  Smoke: add to `security-rls.spec.ts` — "self-auth via direct supabase-js UPDATE is rejected by RLS with code 42501". Cite "RICS Client money handling — segregation of duties; both signatories must be staff of the firm" in the migration comment.
- **Effort:** S (under 30 min). Single-line predicate addition; no Edge Function dependency.

### A-2 · `contractors.approved` / `approved_by` / `approved_at` mutable by non-admin

- **Severity:** CRITICAL
- **File:** `supabase/migrations/00012_rls_policies.sql:131` (FOR ALL `is_pm_or_admin()`; widened by 1i.3)
- **Issue:** Post-1i.3 the payee-setter ≠ release-authoriser gate reads `contractors.approved_by` to compare against the current authoriser. But ANY pm-tier user (admin / accounts / senior_pm / property_manager) can directly UPDATE those columns via supabase-js — including stamping themselves as `approved_by` without an authorising PA. `ContractorsPage.handleSubmit:437-440` explicitly comments "approved is NEVER set directly — only by the PA authorise path" but RLS does not enforce that.
- **Fix:** Column-level GRANT REVOKE pattern (1i.1 C-1 sister):
  ```sql
  REVOKE UPDATE ON public.contractors FROM authenticated;
  GRANT  UPDATE (
    company_name, contact_name, email, phone, address,
    trade_categories, insurance_expiry, gas_safe_number, electrical_approval,
    preferred_order, active, portal_access, rating, notes
  ) ON public.contractors TO authenticated;
  -- `approved`, `approved_by`, `approved_at` are not granted; only the
  -- authorisePayeeSetup branch (running under the admin's authority and
  -- gated by app-side self-auth) can write them. The Edge Function lift
  -- replaces the gating with a server-side check.
  ```
  Note: this makes `bank-detail edit flips approved=false` (ContractorsPage.tsx) require either a service-role bypass or routing through the authorisePayeeSetup branch. Plan-first gate before applying.
- **Effort:** M (under half day). Column-level GRANT pattern; tested in 1i.1; migration + smoke + verification of the ContractorsPage approval flip-path.

### A-3 · `bank_accounts.is_active` / `closed_date` / `rics_designated` mutable by non-admin

- **Severity:** CRITICAL
- **File:** `supabase/migrations/00026_security_hardening.sql:70-74` (post-1i.3 is_pm_or_admin widening cascades through)
- **Issue:** Closure flow uses dual-auth (PA `action_type='close_bank_account'`). Per `BankAccountsTab.tsx:82` `canManageClosure = hasAdminRole(roles)` — UI gate is admin-only. RLS permits any pm-tier user to UPDATE `is_active=false` directly. Same for `rics_designated` (1g.5 direction-gated flow — RLS doesn't enforce direction-gating either). RICS Client money handling client-money-segregation defeated.
- **Fix:** Column-level GRANT REVOKE on `bank_accounts` mirroring A-2 — grant non-admin pm-tier roles UPDATE on cosmetic columns only; reserve `is_active`, `closed_date`, `rics_designated` for admin OR for the financial-rules Edge Function service-role.
- **Effort:** M. Pairs naturally with A-2 in one column-grant migration.
